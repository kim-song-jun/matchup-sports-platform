# Teameet Real-Flow QA Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Teameet의 핵심 기능들을 실제 사용자 흐름 기준으로 검증할 수 있도록, DB 저장성부터 다중 탭/다중 브라우저 실시간 동기화까지 포함한 상세 테스트 시나리오 기준서를 만든다.

**Architecture:** 테스트는 "행동 발생 → UI 반영 → API 응답 → DB 영속화 → 다른 컨텍스트 반영"의 5단계로 검증한다. 자동화는 Playwright를 중심으로 하되, 같은 사용자 다중 탭, 서로 다른 사용자 브라우저 컨텍스트, 필요 시 API 재조회 또는 서버 재시작 검증을 결합한다.

**Tech Stack:** Next.js 15, NestJS 11, Prisma 6, PostgreSQL 16, Redis 7, Socket.IO, Playwright, Jest, Vitest, Docker Compose, Makefile

---

## 0. 문서 목적

이 문서는 단순 QA 체크리스트가 아니다. 아래 세 가지를 동시에 만족하는 실행 기준서다.

1. 기능별 실제 사용자 시나리오를 상세히 고정한다.
2. 이후 Playwright E2E, Playwright MCP, 보조 API/DB 검증 스크립트로 바로 분해할 수 있다.
3. 같은 기능을 단일 브라우저, 다중 탭, 다중 사용자, 새로고침/재접속, 서버 재시작 상황에서 어떤 순서로 검증할지 명확히 한다.

## 1. 테스트 원칙

### 1.1 공통 검증 축

모든 mutation 시나리오는 아래 축을 가능하면 전부 확인한다.

- UI 성공: 사용자에게 성공 상태가 보이는가
- API 성공: 응답이 정상 포맷으로 반환되는가
- DB 영속화: 새로고침, 재접속, 목록 재조회 후에도 유지되는가
- 권한 반영: 허용된 사용자만 보거나 실행할 수 있는가
- 실시간 반영: 다른 탭/다른 사용자 화면에 지연 없이 반영되는가
- 회복력: 네트워크 실패/권한 오류/중복 클릭에서 이상 동작이 없는가

### 1.2 컨텍스트 매트릭스

각 기능은 아래 컨텍스트 중 필요한 조합으로 검증한다.

- `S1`: 단일 탭, 단일 사용자
- `S2`: 같은 사용자, 브라우저 탭 2개
- `S3`: 서로 다른 사용자, 브라우저 컨텍스트 2개
- `S4`: 새로고침/로그아웃/재로그인 후 재검증
- `S5`: API 또는 앱 서버 재시작 후 영속성 재검증

### 1.3 영속성 기준

DB 저장 검증은 아래 우선순위로 수행한다.

1. 화면 재조회로 확인
2. 관련 API 재호출로 확인
3. 필요 시 Playwright 외부 보조 검증으로 DB 직접 조회
4. 서버 재시작 후 동일 데이터가 남아 있는지 확인

## 2. 공통 테스트 환경

### 2.1 기준 URL

- Web: `http://localhost:3003`
- API: `http://localhost:8111/api/v1`
- Swagger: `http://localhost:8111/docs`

### 2.2 기준 페르소나

- `시나로E2E`: 일반 사용자, 개인 매치/탐색 중심
- `팀장오너E2E`: 팀 오너, 팀 매치/승인/운영 중심
- `매니저E2E`: 팀 매니저, 오너가 아닌 운영 권한 사용자
- `일반팀원E2E`: 팀 소속 일반 멤버, 제한 권한 확인용
- `용병호스트E2E`: 용병 모집 호스트
- `관리자E2E`: 관리자 대시보드/분쟁/정산 확인용
- `강사E2E`: 레슨 생성/출석 처리용
- `판매자E2E`: 장터 등록/주문 상태 확인용

### 2.3 기본 준비

- 앱 스택이 실행 중이어야 한다.
- E2E 로그인은 `dev-login` API 또는 로그인 UI를 사용한다.
- 다중 사용자 시나리오는 브라우저 컨텍스트를 분리한다.
- 동일 사용자 다중 탭 시나리오는 같은 storage state를 공유한다.

## 4. 기능별 상세 시나리오

## 4.1 인증 / 세션 / 보호 라우트

### AUTH-001 로그인 후 세션 유지와 새 탭 공유

- 목적: 로그인 상태가 새로고침과 새 탭에서 유지되는지 확인
- 주 사용자: `시나로E2E`
- 컨텍스트: `S1`, `S2`, `S4`
- 사전 조건:
  - 로그인하지 않은 상태에서 시작
- 단계:
  1. `/login`으로 진입한다.
  2. 개발용 로그인 또는 UI 로그인으로 `시나로E2E` 계정에 진입한다.
  3. `/home` 진입을 확인한다.
  4. 현재 탭을 새로고침한다.
  5. 같은 브라우저에서 새 탭을 열고 `/profile`로 이동한다.
- 기대 결과:
  - 첫 로그인 후 홈 화면이 렌더링된다.
  - 새로고침 후 세션이 유지된다.
  - 새 탭에서도 재로그인 없이 보호 페이지가 열린다.
- 실패 검증:
  - 토큰이 손실되어 로그인 페이지로 튕기지 않아야 한다.
  - 보호 페이지에서 빈 화면/무한 로딩이 없어야 한다.
- 자동화 비고:
  - Playwright 기본 인증 헬퍼로 바로 구현 가능

### AUTH-002 비로그인 사용자의 보호 액션 차단

- 목적: 보호 버튼과 보호 라우트가 비로그인 사용자에게 안전하게 동작하는지 확인
- 주 사용자: anonymous
- 컨텍스트: `S1`
- 대상 기능:
  - 팀 상세의 연락하기
  - 용병 지원
  - 채팅 진입
  - 알림 페이지
  - 프로필 페이지
- 단계:
  1. 비로그인 상태로 각 진입점을 연다.
  2. CTA 버튼을 클릭한다.
- 기대 결과:
  - 로그인 유도 또는 로그인 페이지 리다이렉트가 일관된다.
  - `alert()` 같은 임시 피드백이 없어야 한다.
  - 보호되지 않은 API 호출이 발생하지 않아야 한다.

### AUTH-003 관리자 페이지 권한 가드

- 목적: 관리자 전용 페이지가 일반 사용자에게 노출되지 않는지 확인
- 주 사용자: `시나로E2E`, `관리자E2E`
- 컨텍스트: `S1`, `S3`
- 단계:
  1. 일반 사용자로 `/admin/dashboard`에 진입한다.
  2. 관리자 사용자로 같은 페이지에 진입한다.
- 기대 결과:
  - 일반 사용자는 접근 거부 또는 안전한 리다이렉트를 받는다.
  - 관리자는 대시보드, 통계, 리스트 데이터를 볼 수 있다.

  - 캐시/재요청 과정에서 로딩과 완료 상태가 모두 정상이다.

## 4.3 개인 매치

### MATCH-001 매치 생성 후 목록/상세/새로고침 반영

- 목적: 개인 매치 생성이 실제 저장되고 여러 화면에서 일관되게 보이는지 확인
- 주 사용자: `시나로E2E`
- 컨텍스트: `S1`, `S2`, `S4`, `S5`
- 사전 조건:
  - 생성 가능한 사용자 프로필이 준비되어 있어야 한다.
- 단계:
  1. `/matches/new`에서 새 매치를 생성한다.
  2. 성공 후 상세 또는 목록으로 이동한다.
  3. 생성한 매치가 `/matches`, `/home`, `/my/matches`에 보이는지 확인한다.
  4. 같은 사용자 새 탭에서 `/matches/:id`를 연다.
  5. 기존 탭과 새 탭을 모두 새로고침한다.
  6. 가능하면 API 또는 앱 컨테이너 재시작 후 다시 조회한다.
- 기대 결과:
  - 생성 직후 상세 페이지 정보가 입력값과 일치한다.
  - 목록과 내 매치 화면에 반영된다.
  - 새로고침 후에도 사라지지 않는다.
  - 서버 재시작 후에도 남아 있으면 DB 영속성이 확인된다.
- DB 검증 포인트:
  - 생성 시각, 모집 상태, 참가 인원, 비용, 장소가 유지되는지 확인

### MATCH-002 다른 사용자의 참가와 참가 인원 동기화

- 목적: 참가 액션이 호스트와 참가자 양쪽에 즉시 반영되는지 확인
- 주 사용자: `시나로E2E`(호스트), 보조 사용자 임의 일반 계정
- 컨텍스트: `S3`, `S4`
- 단계:
  1. 호스트 컨텍스트에서 매치를 연다.
  2. 다른 사용자 컨텍스트에서 같은 매치 상세로 진입한다.
  3. 참가자가 참가 버튼을 누른다.
  4. 참가 완료 후 참가자 컨텍스트의 상태 변화를 본다.
  5. 호스트 컨텍스트를 새로고침하거나 실시간 반영을 관찰한다.
- 기대 결과:
  - 참가자에게 중복 신청이 허용되지 않는다.
  - 참가 인원 수가 증가한다.
  - 호스트/참가자 양쪽 모두 현재 상태를 정확히 본다.
- 실패 검증:
  - 정원이 가득 찬 경우 차단되어야 한다.
  - 비로그인 사용자는 참가를 완료할 수 없어야 한다.

### MATCH-003 수정 / 취소 / 종료 상태 반영

- 목적: 호스트의 상태 변경이 상세/목록/내 기록에 일관되게 반영되는지 확인
- 주 사용자: `시나로E2E`
- 컨텍스트: `S1`, `S2`, `S4`
- 단계:
  1. 본인이 만든 매치를 연다.
  2. 수정 페이지에서 일정 또는 설명을 바꾼다.
  3. 다시 상세로 돌아와 값이 바뀌었는지 확인한다.
  4. 상태를 마감 또는 취소로 바꿀 수 있다면 실행한다.
  5. 목록과 내 기록에서 변경 상태를 확인한다.

## 4.5 팀 매치

- 주 사용자: `팀장오너E2E` 또는 `매니저E2E`

### TM-002 상대 팀 신청과 상호 확인

- 목적: 신청 팀과 호스트 팀이 같은 데이터를 서로 다른 시점에서 정확히 보는지 확인
- 주 사용자: `팀장오너E2E`, 보조 사용자 팀 계정
- 컨텍스트: `S3`, `S4`
- 단계:
  1. 호스트 팀이 팀 매치를 생성한다.
  2. 다른 사용자 컨텍스트에서 해당 모집글로 이동한다.
  3. 신청 시 어느 팀으로 신청할지 선택한다.
  4. 신청 후 신청 사용자 쪽에서 상태를 확인한다.
  5. 호스트 쪽에서 신청 목록 또는 상호 확인 화면을 연다.
- 기대 결과:
  - 신청 팀 선택이 누락되지 않는다.
  - 호스트는 신청 팀 목록을 볼 수 있다.
  - 신청자는 자신의 상태를 볼 수 있다.

### TM-003 승인 / 거절 후 상태 동기화와 알림 반영

- 목적: 호스트의 승인/거절이 신청자와 알림 시스템에 정확히 반영되는지 확인
- 주 사용자: 호스트 팀, 신청 팀
- 컨텍스트: `S3`, `S4`
- 단계:
  1. 호스트가 신청을 승인한다.
  2. 신청자 쪽 상세/내역/알림 화면을 확인한다.
  3. 필요 시 다시 거절 케이스를 별도로 재현한다.
- 기대 결과:
  - 상태가 `pending -> approved` 또는 `rejected`로 전환된다.
  - 알림이 생성되고 읽기 전 상태가 노출된다.
  - 새로고침 후에도 결과가 유지된다.

### TM-004 도착 인증 / 점수 입력 / 경기 후 평가

- 목적: 경기 당일 이후 단계가 단절 없이 이어지는지 확인
- 주 사용자: 양 팀 대표
- 컨텍스트: `S3`

  1. 양 팀이 도착 인증 페이지에 진입한다.
  2. 점수 입력 및 결과 제출을 수행한다.
  3. 상호 평가 화면으로 이어지는지 확인한다.
  4. 평가 제출 후 경기 상태와 후속 데이터가 반영되는지 확인한다.
- 기대 결과:
  - 순서가 꼬이지 않아야 한다.
  - 이미 완료한 단계는 중복 제출이 제한되어야 한다.

## 5. Playwright 구현 전략

### 5.1 다중 사용자

- 브라우저 컨텍스트를 2개 이상 연다.
- 각 컨텍스트에 서로 다른 페르소나 토큰을 주입한다.

- 같은 컨텍스트에서 페이지를 2개 연다.
- 한 탭에서 변경을 발생시킨 뒤 다른 탭에서 badge/list/detail 변화를 확인한다.

- mutation 직후 새로고침
- 목록 페이지로 이탈 후 재진입
- 새 컨텍스트에서 재조회
- 필요 시 앱/API 재시작 후 동일 데이터 재확인
